## 一、核心业务模型

### 1.1 用户身份体系

**三种身份互相独立，注册时选定，不可切换。** 玩家账号不能发店家局，店家账号不能以玩家身份组局，管理员由系统内置。

| 身份 | 说明 | 注册方式 |
|------|------|---------|
| **玩家** | 参与或发起拼车的普通用户 | 手机号+密码注册，无需其他资质 |
| **店家** | 经营店铺，可发布店家局 | 注册时选择"我是店家" → 创建新店铺或搜索加入已有店铺 |
| **管理员** | 系统管理方 | 内置账号，不可公开注册 |

**JWT 携带角色信息（player / shop / admin）**，后端通过拦截器做接口级权限校验：
- `/api/player/**` — 仅玩家角色
- `/api/shop/**` — 店家账号可访问；涉及具体店铺资源（成员管理、剧本库、拼车）时校验 shop_member 归属
- `/api/admin/**` — 仅管理员角色

### 1.2 店铺体系

**店铺角色权限：** 店长 > 管理员 > 普通成员。**管理员能做的店长都能做**，反之不成立。

| 角色 | 权限                                 | 说明 |
|------|------------------------------------|------|
| **店长** | 设置成员权限、管理店铺剧本库、发布店家局、指派 DM、添加/移除成员 | 店铺创建者自动成为店长，拥有全部权限 |
| **管理员** | 管理店铺剧本库、发布店家局、指派 DM、添加/移除成员        | 可独立经营，但不能提升/降级其他成员的权限 |
| **普通成员** | 可被指派为 DM                           | 不能发布店家局，不能管理店铺 |

**店铺信息字段：**
- 店铺名称、店铺地址、联系电话、Logo、封面图、简介、营业时间

**加入店铺流程（Demo）：**
1. 店家用户搜索店铺名称 → 查看店铺详情
2. 点击"申请加入" → 系统自动通过，直接成为普通成员
3. 店长可在成员管理中将其提升为管理员

> 注：玩家角色不能加入店铺。仅店家角色用户可申请加入已有店铺或以店长身份创建店铺。

**扩展功能：** 店铺申请审核（店长可查看申请、通过/拒绝）、店铺成员主动邀请

### 1.3 剧本体系

**系统剧本库**（管理员维护）：
- 预设 30-50 个市面热门剧本
- 字段：名称、类型、难度、人数范围、参考时长、角色列表、封面图、价格参考

**店铺剧本库：**
- 店家从系统剧本库中选择本店能完成的剧本加入
- 店长和管理员均可管理
- 发布店家局时仅能从店铺剧本库中选择

**扩展功能：** 店家提交新剧本申请 → 管理员审核入库

### 1.4 拼车双模式

**核心概念：** 拼车分为玩家局和店家局两种模式，区别在于发布方身份、DM来源、支付流向和匹配方式。DM（主持人）是拼车内的一个角色，不是独立的用户身份。

| 维度 | 玩家局 | 店家局 |
|------|--------|--------|
| 发布方 | 玩家（默认担任DM），成团前可转让DM | 店家成员（店长或管理员） |
| DM | 发布人默认，成团前可转让给其他成员 | 店家从店铺成员中指派（发布人或任意成员均可） |
| **DM转让对支付的影响** | 释放资金时以当前 `car_pool.dm_id` 为准计算收款方。转让 DM 后，未释放订单的 payee_id 按新 DM 更新 | 店家局 DM 由店铺指派，不存在转让场景 |
| 支付对象 | DM（平台→DM） | 店家（平台→店铺） |
| 匹配方式 | 双向推荐（系统匹配推荐+发布人发布） | 玩家单向浏览选择，系统不做推荐 |
| 剧本来源 | 系统剧本库 | 店铺剧本库 |

**拼车状态机：**
```
OPEN(0) ↔ FULL(1) → COMPLETED(2) → FINISHED(3)
                     ↘ CANCELLED(4)    ↗
```

| 状态 | 含义 | 触发条件 | 触发动作 |
|------|------|---------|---------|
| OPEN(0) | 开放招募 | 发布成功 | — |
| FULL(1) | 人数满员 | current_members == max_members | 自动切换，通知全体 |
| **COMPLETED(2)** | **拼车成功** | **已满员+有DM+全员确认** | **释放押金，自动创建车费订单，此后退出=跳车** |
| **FINISHED(3)** | **剧本杀完成** | **时间围栏+确认机制** | **释放剩余车费，开启评价入口，群聊解散** |
| CANCELLED(4) | 已取消 | 发布人手动取消（仅COMPLETED前）或超时自动取消 | 全额退还已支付金额 |

**确认机制（成员调用 confirm 接口）：**
COMPLETED 和 FINISHED 需要拼车参与者确认，而不是仅由发布人触发：

- 发布人调用 `POST .../complete` 或 `POST .../finish` — **发起确认流程**
- 系统推送确认通知给所有成员（WebSocket + 应用内消息）
- 每位成员调用 `POST .../confirm` 接口提交确认（body: `{ confirmed: true/false }`）
- 确认结果记录在 `pool_member` 表的 `completed_confirmed` / `finished_confirmed` 字段
- 后端校验：同一用户重复调用不重复计数，非成员拒绝调用
- 每次确认后返回当前确认人数/总需确认人数
- **确认参与者：** 仅 `pool_member.status=2(已加入)` 的玩家成员参与确认计数。店家局中的 DM 和店家工作人员不计入确认总人数，但可作为成员提交确认

**确认重新发起：** 若有成员拒绝（confirmed=false），流程不达成。发布人处理后可重新调用 complete/finish 发起新一轮确认，系统重置所有成员的对应确认状态为"未确认"。

**COMPLETED 确认规则（FULL → COMPLETED）：**
- 前提：已满员 + 已指定DM
- 发布人发起 complete → 推送通知给所有成员 → 每人调用 confirm
- **必须全员确认**（不做多数确认）
- 全员确认后 → COMPLETED(2)，释放押金（更新 order.release_status=1）

**FINISHED 确认规则（COMPLETED → FINISHED）：**
- 发布人发起 finish → 推送通知给所有成员 → 每人调用 confirm
- **时间围栏规则：**
  - `end_time` 前发起：必须**全员确认**
  - `end_time` 后发起：**多数确认**（过半数成员确认即可）
- 条件满足后 → FINISHED(3)，释放剩余车费（更新 order.release_status=1），开启评价入口

**超时兜底机制：**
定时任务每 5 分钟扫描超时拼车，规则如下：

| 条件 | 处理方式 | 扫描 SQL 示例 |
|------|---------|--------------|
| OPEN 状态超过 start_time 仍无人加入 | 自动 CANCELLED，退还押金 | `WHERE status=0 AND start_time < NOW() AND current_members = 0` |
| COMPLETED 状态超过 end_time + 48h 仍不足半数确认 | 自动 FINISHED，系统代执行完成 | `WHERE status=2 AND end_time < NOW() - INTERVAL 48 HOUR` |
| 车费订单超过 24h 未支付 | 标记订单逾期（status=4），信用分-10 | `WHERE type=1 AND status=0 AND create_time < NOW() - INTERVAL 24 HOUR` |

**状态流转补充规则：**
- **FULL → OPEN**：有人退出/跳车且不再满员时自动回退
- **COMPLETED → CANCELLED：不允许**（拼车成功后只能正常完成）
- 拼车成功（COMPLETED）前可随时退出，无惩罚；成功后退出视为**跳车**，信用分扣减，群聊权限立即移除
- 跳车不影响已释放资金，押金已在 COMPLETED 时释放
- **满员计算：** `current_members` 仅统计 `pool_member.status=2(已加入)` 的**玩家成员**（即已支付押金的正式成员），待审核和待支付状态不计入。店家局中的 DM 和店家工作人员不计入成团人数
- **加入/支付流程：** 申请加入 → `pool_member.status=0(待审核)`（审核制）/ `1(待支付)`（自动通过）→ 支付押金 → `status=2(已加入)`，`current_members`+1
- **发布人创建流程：** 发布人创建拼车时，系统同步为其创建 `pool_member(status=1(待支付), role=1(发布人))` 记录。`pool_member.role` 为单值枚举，仅表示在拼车中的业务角色（玩家/发布人），DM 身份统一通过 `car_pool.dm_id` 判断（发布人默认兼任 DM 时设置 `car_pool.dm_id=owner_id`）。发布人支付押金后成为正式成员计入 `current_members`；若发布人免押金（如 demo 简化），则创建时直接设为 `status=2(已加入)` 并计入

### 1.6 匹配机制

**玩家局匹配流程：**

```
玩家填写偏好 → 系统匹配已有玩家局
    (剧本类型/时间范围/地点)
                │
                ▼
       匹配引擎打分排序
     (策略链：城市+类型+时间+信用)
                │
                ▼
       WebSocket 推送推荐给玩家
                │
                ▼
       玩家确认加入 ← 审核制需发布人审核
                │
                ▼
           加入成功
```

**店家局匹配：** 玩家浏览筛选店家发布的拼车，自行选择加入，系统不做双向推荐。

**拼车列表接口增强：**
`GET /api/player/pool/list` 和 `GET /api/shop/pool/list` 接口支持以下查询参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `city` | String | 城市筛选 |
| `script_type` | String | 剧本类型（硬核/情感/欢乐/恐怖/机制） |
| `price_min` | Decimal | 最低价格 |
| `price_max` | Decimal | 最高价格 |
| `start_time_after` | DateTime | 最早开始时间 |
| `start_time_before` | DateTime | 最晚开始时间 |
| `page` | Integer | 页码，默认 1 |
| `size` | Integer | 每页条数，默认 20 |
| `recommend` | Boolean | 是否启用匹配排序，默认 false（为 true 时后端调用匹配引擎对结果排序，匹配度高的排前面） |

### 1.7 信用分 vs 评价分

两套独立体系，**均不做排行榜**。

| 维度 | 信用分 | 评价分 |
|------|--------|--------|
| 评估对象 | **玩家**（含玩家局中的DM） | **店家 + DM**（含店家DM和玩家局DM） |
| 覆盖范围 | 店家及其DM**不参与**信用分 | 店家及其DM、玩家局DM **均参与**评价分 |
| 评估内容 | 拼车守约行为（跳车/完成） | 服务质量（玩家评分 1-5） |
| 惩罚 | 信用分过低 → 影响匹配和发车 | 评价分过低 → **冻结DM接单权限** |
| 计算方式 | 系统自动加减 | 玩家主动评分 |

**信用分梯度：**

| 条件 | 变动 |
|------|------|
| 距开团 > 24h 跳车 | -10 |
| 距开团 2-24h 跳车 | -20 |
| 距开团 < 2h 跳车 | -30 |
| 累犯（7 天内 > 2 次） | 额外 -5 |
| 完成拼车 | +5 |

**评价分（1-5 分）：**
- 玩家在**剧本杀完成**后对店家或 DM 评分
- 评价接口参数 `type`：`0` = 评价店家，`1` = 评价 DM
- 评价分过低（如 < 2.0）→ 该 DM **冻结接单权限**一段时间
- 店家评价分影响店铺展示，DM 评价分影响能否被指派

**评价接口：**
- `POST /api/player/review/create` — 提交评价（type: 0-店家 1-DM）
- `GET /api/player/review/my-dm` — 我作为 DM 收到的评价
- `GET /api/shop/review/my` — 我的店铺收到的评价

**评价分最小样本量规则：**
- DM 冻结接单权限的触发条件：**评价数 ≥ 3 条 且 累加平均分 < 2.0**
- 不足 3 条评价的 DM 仅展示评分，不触发冻结
- 简单算术平均，不做加权

**Demo 实现说明：** 不新增 `dm_status` 字段，不做落库冻结。指派 DM 时实时查询该用户收到的评价，若满足"≥3 条且平均分 < 2.0"则禁止指派，前端展示"该 DM 因评分过低暂不可用"。扩展可后增加 DM 状态字段和自动解冻定时任务。

### 1.8 支付流程（Mock）

**核心原则：** 平台作为支付中介，两阶段释放资金。**押金(deposit)是总费用(price)的一部分，不是额外费用。** 总费用 price = 预付押金 deposit + 剩余车费(price - deposit)。

**资金流向图：**
```
玩家加入拼车
    │
    ▼
① 支付押金 (deposit) → 平台冻结
    │
    ▼
拼车成功 (COMPLETED)
    │
    ├─ 押金释放给收款方（玩家局→DM / 店家局→店铺）
    │
    ▼
② 系统自动创建车费订单 (price - deposit，待支付)
    │
    ├─ 玩家主动支付 → 平台冻结
    │
    ▼
剧本杀完成 (FINISHED)
    │
    └─ 剩余车费释放给收款方（玩家局→DM / 店家局→店铺）
```

**分阶段说明：**

| 阶段 | 触发时机 | 支付/释放 | 金额 | 触发方式 |
|------|---------|-----------|------|---------|
| 押金支付 | 加入拼车时 | 玩家 → 平台（冻结） | deposit | 玩家主动调 order/create 创建+pay |
| 押金释放 | COMPLETED（拼车成功） | 平台 → 收款方 | deposit | 系统自动触发 |
| 车费订单创建 | COMPLETED 达成时 | 系统自动创建车费订单 | price - deposit | 系统自动 |
| 剩余车费支付 | COMPLETED 后 | 玩家 → 平台（冻结） | price - deposit | 玩家主动调 order/create?type=1+pay |
| 剩余车费释放 | FINISHED（剧本杀完成） | 平台 → 收款方 | price - deposit | 系统自动触发 |

**支付异常处理策略：**

| 异常场景 | 处理方式 |
|---------|---------|
| 支付失败（网络/余额不足） | 提示重试，不改变订单状态，最多重试 3 次 |
| 重复支付 | 后端幂等校验（按 order_no），重复回调不重复记账 |
| 车费超时未支付（24h） | 定时任务扫描：标记订单"逾期"，信用分-10 |
| 跳车（COMPLETED 后退出） | 信用分扣减（按时间梯度），保留待支付车费订单。Demo 不做强制扣款，未付车费逾期扣信用分 |
| 取消（COMPLETED 前） | 全额退还所有已支付金额 |

**支付接口复用：**
- 押金和车费复用同一个创建接口，通过 `type` 字段区分：
  - `POST /api/player/order/create` body: `{ poolId, type: 0(押金)/1(车费) }`
  - `POST /api/player/order/pay/{orderNo}` — 模拟支付

**Demo 阶段：** 所有支付为 Mock 模拟，架构预留接口，生产环境接入微信支付。Demo 阶段不做自动代扣，未来可扩展为"签约代扣"模式。

### 1.9 群聊

**生命周期：**
1. **创建：** 拼车发布成功后自动创建群聊，无需手动操作
2. **运营中（OPEN/FULL）：** 所有已加入的成员（玩家 + DM + 店家人员）可在群内聊天
3. **拼车成功后（COMPLETED → FINISHED 间）：** 群聊**保留**，成员可继续交流（安排细节、复盘等）
4. **剧本杀完成后（FINISHED）：** 群聊对成员**自动解散**，聊天入口不再可见。后台保留完整审计日志，管理员可查询历史记录。

**成员权限：**
- 所有已加入成员自动进入群聊
- **退出/跳车的成员立即失去群聊访问权限**，不再能看到消息记录
- 审核制下，待审核状态的申请人不能进入群聊
- DM 在群内身份标识与普通成员区分

**消息类型：**
- Demo 阶段仅支持**文本消息**
- 扩展预留：图片、语音消息

**技术实现：** 基于 WebSocket STOMP 协议，每个拼车一个独立 Chat 频道。
- 发送：`POST /api/player/chat/send` 或 `POST /api/shop/chat/send`
- 订阅：`/topic/pool/{poolId}/chat`
- 发送者身份通过 JWT 解析，无需额外传参

### 1.10 通知（应用内消息）

**消息类型（message.type）：**

| type | 含义 | 触发场景 | 推送对象 |
|------|------|---------|---------|
| 0 | 系统消息 | 申请加入、通过/拒绝、取消等 | 相关用户 |
| 1 | 匹配结果 | 匹配引擎推送推荐 | 被推荐的玩家 |
| 2 | 成团通知 | 满员、拼车成功(COMPLETED) | 拼车全员 |
| 3 | 跳车通知 | 成员跳车 | 拼车全员 |
| 4 | 评价提醒 | 剧本杀完成(FINISHED)后可评价 | 拼车全员 |

**具体的消息触发场景：**

| 场景 | 通知内容 | 接收方 | type |
|------|---------|--------|------|
| 申请加入（审核制） | "xxx 申请加入拼车" | 发布人 | 0 |
| 发布人通过申请 | "您的加入申请已通过" | 申请人 | 0 |
| 发布人拒绝申请 | "您的加入申请已被拒绝" | 申请人 | 0 |
| 匹配推荐 | "推荐了 N 个拼车" | 玩家 | 1 |
| 满员 | "拼车已满员" | 全员 | 2 |
| 拼车成功(COMPLETED) | "拼车已成功，押金已释放" | 全员 | 2 |
| 剧本杀完成(FINISHED) | "剧本杀已完成，请评价" | 全员 | 4 |
| 跳车 | "xxx 已跳车" | 全员 | 3 |
| 拼车取消 | "拼车已取消" | 全员 | 0 |
| 确认提醒 | "请确认拼车完成" | 待确认成员 | 0 |

**技术实现：** Demo 阶段采用"业务事务内直接写 message 表 + 事务后尝试 WebSocket 推送"的简化方案：
1. 业务操作的事务内插入 message 表记录（含 msg_key 幂等键防重复）
2. 事务提交后通过 WebSocket 尝试实时通知接收方（推送失败仅记日志，不阻塞业务）
3. 用户通过 `GET /api/player/message/list` 拉取未读消息作为兜底

## 五、接口总览

**接口设计原则：**
- 路径前缀直接代表身份角色：`/api/player/**`（玩家）、`/api/shop/**`（店家账号；操作具体店铺资源时校验 shop_member 归属）、`/api/admin/**`（管理员）
- 公共认证接口（注册、登录、退出）无角色前缀，统一为 `/api/auth/**`
- 统一响应格式：成功返回 `Result.success(data)`，失败返回 `Result.error(message)`
- 分页接口请求参数含 `page`/`size`，响应 data 为 `PageResult<T>`：`{ records: [], total, page, size }`
- **完成确认流程：** `complete` 接口发起确认，所有成员调用 `confirm` 接口提交确认，满足条件后状态变更（规则见 1.4）

### 5.0 公共认证接口

| 接口 | 说明 |
|------|------|
| `POST /api/auth/register` | 注册（body 含 role: player/shop，注册时选定身份） |
| `POST /api/auth/login` | 登录（返回 JWT Access Token + Refresh Token） |
| `POST /api/auth/logout` | 退出登录（将当前 JWT 加入 Redis 黑名单） |
| `POST /api/auth/refresh` | 刷新 Access Token（携带 Refresh Token 换新） |

| 订阅 `/topic/pool/{poolId}/chat` | 群聊消息 |
| 订阅 `/queue/match/{userId}` | 个人匹配推送 |

### 5.5 关键接口详情

以下为核心接口的请求/响应结构示例（完整参数以 Knife4j 文档为准）：

**认证接口：**

| 接口 | 权限 | 请求体 | 成功响应 | 失败响应 |
|------|------|--------|---------|---------|
| `POST /api/auth/register` | 无（公开） | `{ phone, password, nickname, role:player/shop }` | `{ id, phone, nickname, role }` | 400-参数不合法/手机号已注册 |
| `POST /api/auth/login` | 无（公开） | `{ phone, password }` | `{ accessToken, refreshToken, user }` | 401-账号或密码错误 |
| `POST /api/auth/refresh` | 无（携带RefreshToken） | header: `Authorization: Bearer {refreshToken}` | `{ accessToken, refreshToken }` | 401-RefreshToken无效或已过期 |

**拼车接口：**

| 接口 | 权限 | 前置条件 | 成功响应 | 失败响应 |
|------|------|---------|---------|---------|
| `POST .../pool/create` | 玩家/店家 | 玩家局需选系统剧本 | `{ id, status:0 }` | 400-参数不合法 |
| `POST .../pool/{id}/join` | 玩家 | 拼车状态=OPEN，未满员 | `{ memberId, status:0/1 }` | 400-已加入/已满员 |
| `POST .../pool/{id}/leave` | 玩家/成员 | COMPLETED前可退出 | 无返回体 | 400-COMPLETED后跳车需特殊处理 |
| `POST .../pool/{id}/complete` | 发布人 | 状态=FULL，有DM | `{ poolId, confirmedCount, totalCount }` | 400-未满员/无DM |
| `POST .../pool/{id}/confirm` | 成员 | 有进行中的确认流程 | `{ confirmedCount, totalCount, completed:true/false }` | 400-非成员/重复确认 |
| `POST .../pool/{id}/finish` | 发布人 | 状态=COMPLETED | `{ poolId, confirmedCount, totalCount }` | 400-拼车未成功 |

**订单接口：**

| 接口 | 权限 | 请求体 | 成功响应 | 失败响应 |
|------|------|--------|---------|---------|
| `POST .../order/create` | 玩家/成员 | `{ poolId, type:0(押金)/1(车费) }` | `{ orderNo, amount, status:0 }` | 400-重复创建/参数不合法 |
| `POST .../order/pay/{orderNo}` | 订单所属用户 | 无body | `{ orderNo, status:1 }` | 400-订单不存在/已支付 |
| `GET .../order/my` | 用户 | query: `?page&size` | `{ records: [...], total, page, size }` | — |

**订单接口说明：**
- 押金订单仅在成功加入拼车（pool_member 创建）后创建；车费订单仅在拼车成功（COMPLETED）后创建
- 创建订单前按 `user_id + pool_id + type` 查重，防止同一用户对同一拼车重复创建同类型订单
- 资金释放操作仅处理 `status=1(已支付) AND release_status=0(未释放)` 的订单，重复触发直接跳过

**评价接口：**

| 接口 | 权限 | 请求体 | 成功响应 | 失败响应 |
|------|------|--------|---------|---------|
| `POST .../review/create` | 玩家 | `{ poolId, targetId, type:0(店家)/1(DM), score(1-5), content? }` | `{ id }` | 400-已评价过/拼车未完成 |

## 七、附录：前端设计要点

### 7.1 前端定位（暂定）

当前阶段优先建设 **Web Demo**，目标是将已完成的后端接口链路可视化跑通，覆盖玩家、店家、管理员三类角色的核心业务流程。小程序/H5 可作为后续扩展端，当前不优先实现。

### 7.1.1 品牌与视觉基调

- 品牌名：`JuPin 聚拼`
- 首页定位语：`玩家组局、店家开局，一站式剧本杀拼车`
- 面向人群：20-35 岁剧本杀玩家与店家从业者
- 视觉方向：年轻化消费产品
- 未登录首页可偏沉浸氛围，突出剧本杀与拼局感
- 登录后的业务页面保持清爽、易读、操作明确，优先保障联调和测试效率
- 首页第一屏不重点突出信用分/评价机制，避免信息过重；信用分和评价机制作为业务细节在拼车详情、我的页面或后续展示材料中体现

### 7.1.2 图片资源策略（V0）

- V0 使用静态占位图或生成图提升消费产品观感
- 未登录首页使用一张剧本杀氛围主视觉图
- 店铺无图时使用默认店铺封面
- 剧本无图时使用默认剧本封面
- V0 不做图片上传功能
- 图片 URL 可来自初始化数据，也可由前端默认资源兜底

### 7.2 技术栈

- **框架：** Vue 3 + Vite + TypeScript
- **路由：** Vue Router
- **状态管理：** Pinia

- **UI 组件：** Element Plus
- **本地开发代理：** Vite devServer proxy，将 `/api` 转发到后端 `http://localhost:8080`
- **生产部署：** Nginx 托管 `dist` 静态文件，并反向代理 `/api` 到 Spring Boot 服务

### 7.4 页面与路由原则

- 未登录首页提供分角色入口：`我是玩家`、`我是店家`
- 点击角色入口后直接进入对应身份的登录/注册页，不额外增加角色介绍页
- 玩家/店家注册页分离，注册身份由入口固定，注册页不再二次选择角色
- 登录时后端校验账号角色与入口角色一致，不一致则拒绝登录
- V0 使用 `localStorage` 保存 token 与用户基础信息，刷新页面后保持登录态
- 页面切换：玩家界面前置 `/player/`，店家界面前置 `/shop/`，管理后台前置 `/admin/`
- 账号身份注册时固定（玩家/店家），不可切换；玩家和店家使用独立路由入口
- 路由守卫校验登录态与角色：未登录跳转登录页；玩家不能访问店家/管理员页面；店家不能访问玩家/管理员页面
- 前端按钮显示基于状态常量判断；后端仍必须做最终权限和状态校验
- 玩家端采用左侧导航切换，一级页面包括：`玩家局列表`、`店铺列表`、`我的`
- `玩家局列表` 仅展示玩家发起的拼车局；V0 阶段支持玩家在 `我的` 页面手动维护偏好，玩家局推荐/排序基于玩家偏好与拼车信息计算；店家局不混入该列表
- `店铺列表` 展示店铺卡片，点击进入店铺主页，在店铺主页中查看该店铺信息、店铺剧本和店家局
- `我的` 聚合个人相关内容，包括我的拼车、我的订单、待支付、待确认、待评价、玩家偏好编辑等

### 7.5.1 玩家偏好与推荐规则（V0）

玩家偏好在玩家端 `我的` 页面中手动维护，V0 阶段不做注册后强制填写。

| 偏好字段 | V0 规则 |
|---------|--------|
| 常驻城市 | 单选，仅精确到城市，不做商圈/区县 |
| 偏好剧本类型 | 单选，V0 阶段只维护一个主要偏好类型；后续可扩展为多选 OR 匹配 |
| 可接受价格区间 | 最低价/最高价 |
| 常玩时间段 | 可选，用于推荐排序；V0 可先简化为工作日晚上/周末下午/周末晚上 |
| 可接受人数范围 | 可选，用于推荐排序 |

玩家局推荐仅作用于玩家发起的拼车局，不包含店家局。偏好只影响排序，不作为硬过滤条件：玩家局列表仍展示全部可见玩家局，符合偏好的局排在前面。V0 推荐采用可解释的加权排序，不追求复杂算法：

```text
推荐分 = 城市匹配分 + 剧本类型匹配分 + 价格匹配分 + 时间匹配分 + 人数匹配分 + 发布人信用分权重
```

若玩家未维护偏好，则玩家局列表退化为默认排序：同城市优先、开局时间更近优先、人数接近满员优先、创建时间较新优先。

### 7.5.2 玩家局卡片展示规则（V0）

玩家局列表使用卡片式列表展示。卡片字段保持简洁，优先帮助玩家快速判断是否加入。

| 字段 | 展示规则 |
|------|---------|
| 剧本名 | 主标题 |
| 剧本类型 | 短标签，如“硬核”“情感”“机制” |
| 城市 | 短文本 |
| 开始时间 | 展示日期 + 时间 |
| 人数进度 | `当前人数 / 最大人数` |
| 价格 | 同时展示总价与人均车费 |
| 发布人 | 展示昵称 |
| 拼车状态 | 使用状态标签 |
| 推荐理由 | 使用短标签，不展示长句 |

推荐理由短标签示例：

```text
同城
偏好类型
预算内
快满员
时间近
高信用
```

### 7.5.3 玩家局创建价格规则（V0）

玩家创建玩家局时采用 AA 价格模型：

- 发布人填写人均总费用，对应后端 `car_pool.price`
- 系统根据 `max_members` 计算整场预计总价，用于前端展示
- 固定押金 = 人均总费用的 20%，对应后端 `car_pool.deposit`
- 前端创建页展示实时计算结果：人均总费用、整场预计总价、押金、尾款车费
- 后端创建订单时必须使用同一计算口径，前端计算仅用于展示

计算公式：

```text
整场预计总价 = 人均总费用 * 最大人数
押金 = 人均总费用 * 20%
尾款车费 = 人均总费用 - 押金
```

### 7.5.4 玩家局加入方式（V0）

玩家创建玩家局时保留两种加入方式：

| 加入方式 | 规则 |
|---------|------|
| 自动通过 | 默认选项。玩家申请加入后直接进入待支付状态，支付押金后成为正式成员 |
| 审核制 | 玩家申请加入后进入待审核状态，发布人同意后进入待支付状态，支付押金后成为正式成员 |

前端创建页默认选择 `自动通过`，但允许发布人切换为 `审核制`。玩家端 `我的` 页面需要展示发布人待审核事项。

### 7.5.5 店铺主页与店家局展示（V0）

- 玩家从 `店铺列表` 点击店铺卡片进入店铺主页
- 店铺主页展示店铺基础信息、店铺剧本、该店发布的店家局
- 店家局列表展示该店铺下的所有店家局，包括开放、满员、拼车成功、已完成、已取消等状态
- V0 阶段店家局列表不做排序，按后端返回顺序展示
- 玩家是否能加入由店家局状态与成员状态决定，前端只展示可操作按钮，后端做最终校验

店铺列表卡片展示字段：

| 字段 | V0 规则 |
|------|--------|
| 店铺名 | 必须展示 |
| 城市 | 必须展示 |
| 地址 | 必须展示 |
| 营业时间 | 有则展示 |
| 店铺简介 | 展示一行摘要 |
| Logo/封面图 | 有则展示；无图使用默认占位图 |
| 店铺评分 | 预留展示位，V0 暂时显示“暂无评分” |

V0 店铺列表不展示当前店家局数量，避免依赖额外聚合接口。

### 7.5.6 店家端工作台与店铺绑定（V0）

- 店家登录后进入工作台总览
- V0 工作台总览中的综合数据只做占位展示，不做真实统计
- 占位指标可包括：今日开局数、开放中店家局、待审核成员、待确认完成、收到评价均分
- 店家端 V0 不支持创建店铺和申请加入店铺
- V0 默认店家账号已绑定店铺，绑定关系通过初始化数据或后端已有流程提前准备
- 店家端重点支持已绑定店铺下的店铺信息查看、店铺剧本库查看、店家局创建、成员审核/管理、DM 指派

### 7.5.7 店家局创建规则（V0）

- 店家创建店家局时从本店店铺剧本库选择剧本
- 店家局价格同样填写人均总费用，对应后端 `car_pool.price`
- 固定押金规则与玩家局一致：押金 = 人均总费用的 20%
- 店家创建店家局时必须立即指定 DM
- 店家局创建时后端强制 `join_type=1(自动通过)`，不接受审核制
- 可选 DM 范围为当前店铺成员，具体可指派资格由后端校验
- 店长/管理员可以指定自己为 DM
- 普通店铺成员不能创建店家局，但可以被店长/管理员指定为 DM
- 前端创建页需要展示人均总费用、整场预计总价、押金、尾款车费

### 7.5.8 玩家加入店家局规则（V0）

- 玩家加入店家局默认自动通过
- 玩家点击加入后直接进入待支付状态
- 玩家支付押金后成为正式成员，并计入 `current_members`
- 店家局满员后不支持候补，前端隐藏加入按钮，后端拒绝加入请求
- V0 店家端成员审核主要用于玩家局审核制或后续扩展场景，店家局不走审核

### 7.5.9 玩家端“我的”页面结构（V0）

玩家端 `我的` 页面采用 Tab 切换结构，避免左侧一级导航过多：

| Tab | 内容 |

| 我的偏好 | 城市、偏好剧本类型、价格区间、常玩时间段、可接受人数范围 |
| 我的拼车 | 我发布的玩家局、我加入的玩家局/店家局 |
| 我的订单 | 押金订单、车费订单、支付状态、退款状态 |
| 待办事项 | 待支付、待确认、待评价、作为发布人的待审核申请 |

待办事项采用统一列表展示，每条待办包含：

| 字段 | 说明 |
|------|------|
| 待办类型 | 待支付 / 待确认 / 待评价 / 待审核 |
| 关联拼车 | 展示剧本名、开局时间、拼车类型 |

V0 阶段待办列表可由前端基于我的拼车、我的订单等接口结果组装；后续可扩展为后端统一待办接口。

确认类待办前端文案区分：

|---------|---------|

### 7.5.10 评价流程（V0）

- 评价不强制，不阻断用户继续使用系统
- 店家局完成后，前端引导玩家评价店铺和 DM，但允许用户跳过
- 玩家局完成后，前端只引导玩家评价当前 DM
- 待评价事项只作为提醒入口，不作为必须完成的任务
- 后端继续限制同一用户对同一拼车同一评价类型只能评价一次

### 7.5.11 群聊与通知（V0）

- V0 做最简群聊，只支持文本消息收发
- 群聊入口放在拼车详情页，仅已加入该拼车的成员可见
- 玩家正式加入拼车后即可进入群聊
- 拼车进入 `FINISHED(3)` 或 `CANCELLED(4)` 后群聊保留历史记录，但不允许继续发送新消息
- 群聊消息展示发送人、发送时间、文本内容
- V0 不做图片、语音、撤回、已读回执、在线状态
- 通知列表进入 V0，用于展示成团提醒、支付提醒、确认提醒、评价提醒等系统通知
- WebSocket 实时推送作为增强能力；若实时推送不稳定，前端通过通知列表拉取作为兜底
- 群聊和通知权限由后端校验，前端只做入口显示

### 7.6 接口封装约定

- 所有请求统一经过 Axios 实例，自动携带 `Authorization: Bearer <token>`
- 统一处理后端响应 `Result<T>`：成功取 `data`，失败展示 `msg`
- 所有列表接口返回 `Result<PageResult<T>>`，前端从 `data.records` 取列表、`data.total` 取总数
- 模拟支付：点击支付按钮直接调用后端支付接口，不接真实第三方支付
- 前端维护与后端一致的状态字典，避免页面中散落魔法数字

### 7.8 主要风险

- **角色权限复杂：** 玩家、店家、管理员页面和操作不同，前端路由守卫和后端接口权限需要同时兜底
- **状态流转复杂：** 拼车、成员、订单、支付、退款、评价状态相互影响，按钮显示条件必须和后端状态机一致
- **接口变更风险：** 后端仍在补强，前端必须通过 `api/` 层隔离接口变更
- **WebSocket 风险：** 聊天和通知涉及连接、断线重连、token 鉴权，Demo 阶段先保证基础收发与消息列表兜底
- **部署风险：** 需要处理跨域、环境变量、Nginx 代理、数据库初始化和日志排查

## 八、安全设计

### 8.1 密码安全

- **加密算法：** BCrypt（Spring Security BCryptPasswordEncoder），cost factor = 10
- **传输安全：** 登录/注册接口强制 HTTPS（Demo 阶段可放宽）
- **禁止行为：** 明文存储密码、前端硬编码密钥

### 8.2 JWT 策略

| 项目 | 策略 |
|------|------|
| Access Token 有效期 | 2 小时 |
| Refresh Token 有效期 | 7 天 |
| 刷新接口 | `POST /api/auth/refresh`（携带 Refresh Token 换新 Access Token + 新 Refresh Token） |
| Refresh Token 轮换 | 每次刷新发放新 Refresh Token，旧 Refresh Token 立即失效 |
| 服务端存储 | Refresh Token 的 jti 存储于 Redis，支持主动吊销 |
| 退出登录 | 当前 Access Token 的 jti + 对应 Refresh Token 的 jti 加入 Redis 黑名单，过期时间对齐 Token 剩余有效期 |
| 签名算法 | HMAC-SHA256，密钥配置在 application.yml，不同环境使用不同密钥 |

### 8.3 接口防护

| 防护类型 | 规则 |
|---------|------|
| 全局限流 | IP 级别 100 次/分钟，用户级别 30 次/分钟 |
| 敏感接口限流 | 登录、注册、支付接口：5 次/分钟/IP |
| 角色校验 | Spring Security 拦截器按路径前缀校验 JWT 中的角色信息 |
| 请求方法校验 | 严格区分 GET/POST/PUT/DELETE，拒绝非预期方法 |

### 8.4 输入校验

- **原则：** 所有接口参数由后端做完整性校验，前端校验仅用于提升用户体验
- **实现：** Spring Validation（`@Valid` + `@NotBlank`/`@Size`/`@Pattern`）
- **校验规则：**
  - 手机号：正则 `^1[3-9]\\d{9}$`
  - 密码：8-32 位，至少包含字母和数字
  - 昵称：2-20 位，特殊字符过滤
  - 枚举值：白名单校验（如 role 只能为 player/shop，gender 只能为 0/1/2）
- **错误响应：** 统一返回 `400 Bad Request` + 字段级错误信息

### 8.5 其他防护

| 防护类型 | 策略 |
|---------|------|
| SQL 注入 | MyBatis 强制参数化查询（`#{}` 而非 `${}`），禁止 SQL 拼接 |
| XSS | API 响应 Content-Type 固定 `application/json;charset=utf-8`；用户生成内容（昵称、评价、聊天消息）输出时做 HTML 转义，限制长度（昵称 2-20 字，评价 500 字，聊天 1000 字）；敏感词基础过滤 |
| CSRF | 前后端分离架构，Access Token 存储在前端内存或 localStorage，通过 Authorization Header 发送（不自动携带 cookie），已具备防 CSRF 能力，禁用 Spring Security 默认 CSRF 保护 |

## 九、事务与一致性策略

### 9.1 核心原则

| 场景 | 策略 | 说明 |
|------|------|------|
| 单库操作 | Spring `@Transactional`（本地事务） | 同一数据源内的状态变更 |
| 并发控制 | Redisson 分布式锁 | 加入拼车、完成确认等并发敏感操作 |

### 9.2 消息投递方案

Demo 阶段采用简化方案：业务逻辑中直接写 message 表，WebSocket 实时推送作为增强（推送失败不影响主流程）。

| 步骤 | 说明 |
|------|------|
| ① 业务操作 | 本地事务内更新业务状态（订单、信用分等） |
| ② 写入消息 | 同一事务内插入 message 表记录（含 stable msg_key，通过唯一约束防重复） |
| ③ WebSocket 推送 | 事务提交后尝试 WebSocket 实时通知接收方，推送失败仅记日志，不阻塞业务 |
| ④ 用户拉取 | 未收到实时推送的用户通过 `GET /message/list` 拉取未读消息 |

**幂等键生成规则（稳定业务键，不含时间戳）：** `msg_key = eventType + "_" + sourceId + "_" + userId`
示例：`"pool_full_8_3"`（拼车 8 满员通知用户 3），`"pool_complete_8_3"`（拼车 8 完成确认通知用户 3），`"order_overdue_42_15"`（订单 42 逾期通知用户 15）

### 9.3 关键操作事务边界

| 操作场景 | 事务策略 | 说明 |
|---------|---------|------|
| 加入拼车 + 押金创建 | 本地事务 | 同一事务内插入 pool_member（status=待支付）+ 创建订单 |
| 支付回调 + 订单更新 | 本地事务 | 事务内更新订单状态 + 写 message 表；事务提交后推 WebSocket |
| 信用分扣减 + 日志写入 | 本地事务 | 扣分和 credit_log 写入在同一事务 |
| 完成确认 + 计数 + 状态变更 | 本地事务 + 分布式锁 | Redisson 锁住 poolId，防止并发确认重复计数 |
| 状态变更 + 通知写入 | 本地事务 | 事务内更新业务状态 + 写 message 表；事务提交后推 WebSocket |
